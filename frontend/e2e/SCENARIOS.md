# E2E Scenarios

## Real backend (`@real`)

1. `GUEST-BOOKING-SUCCESS-REAL`
   - Гость выбирает тип события, день, свободный слот, вводит данные и подтверждает запись.
   - Ожидается экран успеха с деталями бронирования.

2. `ADMIN-SEES-BOOKING-REAL`
   - После успешного бронирования админ открывает список встреч.
   - Ожидается новая запись в `/admin/bookings`.

3. `NO-DOUBLE-BOOKING-REAL`
   - Повторная попытка создать бронирование на тот же `startUtc`.
   - Ожидается конфликт (`Slot is already booked.`), дубль не появляется в админке.

4. `CROSS-TYPE-OCCUPANCY-REAL`
   - На дне уже занятого времени открывается другой тип события.
   - Ожидается хотя бы один слот со статусом `Занято`.

5. `ADMIN-EVENT-TYPE-CRUD-REAL`
   - Админ создаёт, редактирует и удаляет тип события.
   - Ожидается синхронное отображение в админской и публичной части.

6. `CONFIRM-PARAMS-GUARD-REAL`
   - Переход на `/book/:id/confirm` без параметра `start`.
   - Ожидается сообщение о некорректных параметрах.

7. `BOOKING-FORM-VALIDATION-REAL`
   - Попытка отправить форму с невалидным email.
   - Ожидается клиентская ошибка валидации email.

8. `BOOKING-WINDOW-14D-REAL`
   - Запрос слотов с диапазоном до/после допустимого окна.
   - Ожидается `400 Bad Request`.

## Prism mock (`@mock`)

1. `SMOKE-MOCK-BOOKING-FLOW`
   - Гостевой путь от списка типов событий до экрана успеха.

2. `SMOKE-MOCK-ADMIN-PAGES`
   - Доступность админских страниц бронирований и типов событий.
